## Why

Today the `MarkdownEditor` component has 16 custom Lexical plugins, a 762-line transformer module, and 5 shared helpers ‚Äî and **zero automated tests**. Every feature ship (nested blockquotes, syntax highlighting, list-exit, table keyboard) has been verified by hand in `bun run tauri dev`, which is slow, easy to skip on a "small" change, and leaves regressions free to ride along. Project-level `AGENTS.md` openly admits this: "Frontend (`src/`): no test framework. Verification of changes is `bunx tsc --noEmit` + lint + `tauri dev` for visual smoke tests."

The editor is now the most behavior-dense surface in the codebase. Markdown round-trip correctness, plugin keyboard contracts, and edge cases like "Backspace at the start of a nested blockquote" need to be locked in by tests before the next feature lands ‚Äî otherwise each iteration's hand-verification cost compounds. We do it now, while the editor's structure is fresh and the recent nested-blockquote review surfaced concrete bugs that tests would have caught (e.g. duplicate epilogue in `QUOTE.replace`, false quote-exit on a leading link, trailing-space round-trip drift).

This change is intentionally scoped to **plumbing + a thin proof-of-concept slice**: pick the runner, pick the DOM environment, set up the helpers, write four representative tests that exercise the three testable surface shapes the editor uses today (the transformer shape gets two examples ‚Äî a basic round-trip and a Cork-specific nested variant ‚Äî so the contributor extending coverage sees both flavors). Broad coverage of every plugin and transformer is explicitly out of scope ‚Äî that work will follow in subsequent changes once the foundation is proven.

## What Changes

- **Test runner**: introduce **Vitest 4.1** as the frontend test runner, in **browser mode** so tests execute inside real Chromium driven by Playwright. Wired through `bun run test` / `bun run test:watch` scripts (Vitest runs under Node, invoked by Bun ‚Äî same pattern as `vite`, `tsc`).
- **DOM environment**: real Chromium via `@vitest/browser-playwright`, not `jsdom`. Cork ships in a Tauri WebView (Chromium-class on most platforms), so testing in a real browser is closer to production than jsdom is. The editor's behavior is dense in surfaces (`Selection`, `Range.getBoundingClientRect`, contenteditable, `PointerEvent`) that jsdom either doesn't implement or implements differently ‚Äî Lexical's own jsdom harness ships four polyfills to paper over exactly those gaps, and we'd rather skip that polyfill layer than maintain it.
- **React renderer + interactions + assertions**: use the Vitest-native stack that browser mode is built around ‚Äî `vitest-browser-react` for `render(<Component/>)`, `userEvent` imported from `vitest/browser` for keyboard/click simulation, and `expect.element(‚Ä¶).toBeInTheDocument()` / `.toBeVisible()` as built-in matchers. The Vitest browser-mode docs explicitly position this as the replacement for `@testing-library/react` + `@testing-library/user-event` + `@testing-library/jest-dom` once you're in browser mode; we don't install those three.
- **Vitest config**: a separate `vitest.config.ts` that `mergeConfig`s with `vite.config.ts` so the `@/*` alias, the `@vitejs/plugin-react` pipeline (including `babel-plugin-react-compiler`), and the Tailwind plugin are shared without drift. Test files use `*.spec.ts` / `*.spec.tsx` colocated next to the source they cover (`transformers.ts` ‚Üí `transformers.spec.ts`).
- **Setup file**: a small `vitest.setup.ts` whose only job is `import 'vitest-browser-react'` ‚Äî required by `vitest-browser-react`'s docs so TypeScript picks up its type augmentations (and the browser-mode `page.render` injection). No DOM polyfills, no `expect.extend` ‚Äî those are built in.
- **Shared MarkdownEditor test utilities**: a colocated `__tests__/utils.tsx` exporting `createTestHeadlessEditor()` (for transformer / pure-helper tests, runs purely in JS without mounting a real editor), `renderTestEditor()` (mounts a `LexicalComposer` with Cork's nodes registered via `vitest-browser-react`'s `render`, returns `{editor, screen, user}` for plugin tests), and small state helpers (`$setMarkdown`, `$readMarkdown`). Modelled on Lexical's `packages/lexical/src/__tests__/utils/index.tsx`.
- **Four proof-of-concept tests** covering the three testable surface shapes the editor uses today (with the transformer shape demonstrated by two examples ‚Äî a basic round-trip and a Cork-specific nested-depth variant ‚Äî so the contributor extending coverage sees both flavors):
  1. **Pure helper** ‚Äî `codeBlock.spec.ts`: `$isInsideCodeBlock` returns true under a `CodeNode`, false otherwise (headless editor, no mount).
  2. **Transformer round-trip (basic)** ‚Äî `transformers.spec.ts`: `# Hello\n` ‚Üí editor state ‚Üí markdown SHALL be `# Hello`. (headless editor, no mount).
  3. **Transformer round-trip (Cork-specific nested)** ‚Äî `transformers.quote.spec.ts`: `> > Hello\n` round-trips through the custom QUOTE transformer with the depth-2 nested tree shape Cork defined.
  4. **Plugin keyboard contract** ‚Äî `QuoteExitPlugin.spec.tsx`: rendered `LexicalComposer` + plugin in real Chromium, dispatch `KEY_BACKSPACE_COMMAND` at the start of an empty `QuoteNode`, assert the quote unwraps to a `ParagraphNode`.
- **CI integration**: extend the existing `.github/workflows/ci.yml` `lint` job to (a) install Playwright Chromium with the official headless-CI flag set (`bunx playwright install --with-deps --only-shell chromium`, cached by Playwright version) and (b) run `bun run test` after lint.
- **Lint integration**: extend `.oxlintrc.json` with a `**/*.spec.ts(x)` + `**/__tests__/**` override that relaxes the molecules-layer import restriction so test files can import Lexical fixtures and cross-layer code without fighting the architecture rules. (The runtime rules are unchanged for non-test files.)
- **Documentation**: update `AGENTS.md` (root) ‚Äî replace "no test framework, only `tsc + lint + tauri dev` for verification" with the actual test commands. Add `src/components/molecules/MarkdownEditor/AGENTS.md` section pointing to the test utilities and naming the three surface shapes a contributor should pick from (plus a note that the fourth, live-typing transforms, is reachable with the same helpers but doesn't have a POC yet).
- **Out of scope (explicitly deferred to follow-up changes)**:
  - A POC test for the **plugin live-typing transform** shape (plugins whose behavior is driven by a `registerUpdateListener` rather than a key-press command ‚Äî `QuoteNestingShortcutPlugin`, `CheckListShortcutPlugin`, `FormatShortcutPlugin`). The first follow-up change that touches one of those plugins should add the template for this shape; the helper module is designed to accommodate it (`renderTestEditor` + `editor.update` is enough ‚Äî no new helper needed).
  - Tests for any of the other 15 plugins beyond `QuoteExitPlugin` (e.g. `ListExitPlugin`, `TableKeyboardPlugin`, `CodeBlockHighlightPlugin`, the format-shortcut plugins).
  - Tests for the floating UI plugins (`FloatingFormatToolbarPlugin`, `FloatingLinkEditorPlugin`). Browser mode now makes these tractable (real layout is available), but they bring their own assertion surface (popover position, anchor calculation) that's its own design exercise ‚Äî separated to keep this change small.
  - Tests for any non-editor components (atoms / other molecules / organisms / pages). The editor is the highest-density surface and the proof of the framework; once it's covered, the same patterns extend to the rest of the UI tree.
  - E2E tests against the Tauri binary ‚Äî orthogonal layer, would belong in a separate `tauri-driver` setup.
  - Coverage reporting (`@vitest/coverage-v8`), snapshot testing, and visual regression ‚Äî add when the test surface is broad enough to make the metric meaningful.

## Capabilities

### New Capabilities

- `markdown-editor-testing`: developer-facing contract for what the MarkdownEditor SHALL expose to make its behaviors testable ‚Äî a headless-editor surface for pure transformer / helper tests, a Lexical-command surface for plugin-keyboard tests driven inside a real browser, and a shared test-utility module. Locks in the testability promise so future plugins are written with tests in mind, not bolted on after.

### Modified Capabilities

- _none_

## Impact

- **Code (frontend only)** ‚Äî additions, no behavioral changes to runtime code:
  - **New**: `vitest.config.ts`, `vitest.setup.ts` (root).
  - **New**: `src/components/molecules/MarkdownEditor/__tests__/utils.tsx`, `codeBlock.spec.ts`, `transformers.spec.ts`, `transformers.quote.spec.ts`, `QuoteExitPlugin.spec.tsx`.
  - **Modified**: `package.json` (devDependencies + scripts), `.oxlintrc.json` (test-file override), `.github/workflows/ci.yml` (Playwright install + `bun run test` in lint job), `AGENTS.md` (root), `src/components/molecules/MarkdownEditor/AGENTS.md` (test-utility section).
- **Dependencies** ‚Äî **five** new `devDependencies` only. Versions were verified by installing the proposed stack in a clean worktree (`bun add -D vitest @vitest/browser-playwright playwright vitest-browser-react @lexical/headless@0.45.0` ‚Üí no peer-dep warnings, `bun run vitest run` reaches the Playwright launch step and only fails on the absent browser binary as expected):
  - `vitest` (`4.1.9` exact via `bunfig.toml`'s `exact = true`) ‚Äî consumer code imports the browser-mode API from the `vitest/browser` subpath of this main package. (An internal `@vitest/browser` npm package also exists and is pulled in transitively by `@vitest/browser-playwright`, but consumers never import from it directly.)
  - `@vitest/browser-playwright` (`4.1.9` exact) ‚Äî Playwright provider for browser mode. Its `peerDependencies.vitest` is pinned to the exact same patch as `vitest`'s `peerDependencies."@vitest/browser-playwright"` (mirror pin), so both packages MUST move together; treating either as a caret range is risky if patches release asymmetrically.
  - `playwright` (`1.61.0` exact ‚Äî `1.61.1` is the latest patch but was published 2026-06-23 and is blocked by Cork's `bunfig.toml` `install.minimumReleaseAge = 604800` (7-day cooling-off policy). `1.61.0` (published 2026-06-15, 12 days old) is the current installable patch. The 1.61.1 release notes mention a Node 22.15+ sync-loader fix; Cork's CI invokes `playwright install` via `bunx` (Bun runtime, not Node) so the bug surface doesn't apply here. A follow-up bump to 1.61.1 should land once the age threshold clears (~2026-06-30)) ‚Äî **required** non-optional peer dependency of `@vitest/browser-playwright` (`peerDependenciesMeta.playwright.optional: false`), used via `import('playwright')` inside the provider at runtime. Must be installed explicitly; it is NOT bundled.
  - `vitest-browser-react` (`2.2.0` exact) ‚Äî the Vitest-native React renderer that browser mode is designed around. `peerDependencies` accept `react ^18 || ^19` (Cork's 19.2.7 fits). Reading `src/index.ts` confirmed `import "vitest-browser-react"` auto-registers `page.render` injection and a `beforeEach(cleanup)` hook ‚Äî no additional setup-file boilerplate needed.
  - `@lexical/headless` (`0.45.0` exact ‚Äî matching the rest of the `@lexical/*` family) ‚Äî exposes `createHeadlessEditor` for the transformer / pure-helper tests that don't need to mount a real `LexicalComposer`. Cork's existing dependency list does not include it; the worktree-install check surfaced this gap.
  - **Not installed** (and explicitly documented as not needed in browser mode): `jsdom`, `@testing-library/react`, `@testing-library/user-event`, `@testing-library/jest-dom`. Their roles are filled by `vitest-browser-react`, `userEvent` from `vitest/browser`, and the 27 built-in `expect.element(‚Ä¶)` matchers (jest-dom-compatible names: `toBeInTheDocument`, `toBeVisible`, `toHaveTextContent`, `toHaveClass`, `toHaveAttribute`, `toBeChecked`, `toHaveFocus`, ... ‚Äî verified by reading `packages/browser/src/client/tester/expect/index.ts` in vitest@v4.1.9). `@lexical/headless` transitively brings in `happy-dom@^20`, but it's an internal implementation detail of that package ‚Äî we never import from it.
- **APIs / data on disk**: none. No production code is modified.
- **Backend (Rust)**: no impact.
- **CI runtime cost**:
  - Cold: ~30‚Äì60 s for `bunx playwright install --with-deps --only-shell chromium` on a fresh runner. Cached by Playwright version after the first run.
  - Per-push (cache hit): ~10‚Äì15 s for browser startup + the four POC tests.
  - Negligible vs. the existing `tauri build` job's ~10+ minutes.
- **Risk surface**:
  - **React Compiler interaction**: `vite.config.ts` runs `babel-plugin-react-compiler` and Vitest will inherit it via shared config. Design records why we keep it enabled in tests (parity with production).
  - **Bun + Vitest**: Vitest MUST be invoked via `bun run test` (the local `vitest` bin's `#!/usr/bin/env node` shebang dispatches to Node). Do NOT use `bunx vitest` ‚Äî `bunx` runs the binary's JS under Bun's own runtime (ignoring the shebang), and Vitest 4 has no documented Bun-runtime support. The test code itself runs in the Playwright-driven Chromium page.
  - **Tauri WebView vs Chromium parity**: Tauri uses WKWebView (WebKit) on macOS, WebView2 (Chromium) on Windows, and WebKitGTK (WebKit) on Linux. Our tests run only in Chromium. WebKit-specific bugs that only manifest on macOS or Linux Tauri would not be caught here; explicitly accepted ‚Äî the cost of also running WebKit in CI isn't justified at four tests, and Lexical's own E2E suite covers WebKit separately so the framework-level WebKit gaps are well-mapped already.
  - **Playwright install size in CI**: the `--only-shell` Chromium variant is roughly half the size of the full Chromium binary (Playwright's `/docs/browsers` page does not quote exact MB figures), optionally cached by Playwright version. Not a blocker.
