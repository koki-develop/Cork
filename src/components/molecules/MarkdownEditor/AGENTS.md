# MarkdownEditor (`src/components/molecules/MarkdownEditor/`)

WYSIWYG Markdown editor (Lexical) for the task body. Seeded once from `initialValue` (uncontrolled), emits a Markdown string via `onChange`, forwards `onBlur`. Inline Markdown shortcuts + undo/redo, plus a selection-triggered floating toolbar (no static toolbar / slash commands).

`onOpenLink` is a required prop (Tauri side-effect injected from the page per the org/molecule contract).

## Custom plugins

Sixteen custom plugins bundled in this folder (Lexical's own `AutoLinkPlugin` / `ListPlugin` / `CheckListPlugin` / `TablePlugin` / `HorizontalRulePlugin` are listed where relevant below):

- **`LinkOpenPlugin`** — click a link → `onOpenLink(url)`, wired to the system browser; handles both `[text](url)` links and bare URLs.
- **`ListTabIndentationPlugin`** — Tab / Shift+Tab indent-outdent inside list items only.
- **`ListExitPlugin`** — fixes two delete-shaped Lexical defaults around lists, both at `COMMAND_PRIORITY_LOW` (ahead of rich-text's `EDITOR` handlers, behind `TableKeyboard`'s `CRITICAL`). **(A) Collapsed-at-start backward delete** (Backspace / Cmd+Backspace / Option+Backspace) at the START of a list item exits the list instead of folding the item's content into the line above (which the default merge would read as "marker silently glued onto the previous item"). Hooks all three backward channels — `KEY_BACKSPACE_COMMAND`, `DELETE_LINE_COMMAND` (backward), `DELETE_WORD_COMMAND` (backward). Three cases through one entry point: (1) **empty item** — dispatches `INSERT_PARAGRAPH_COMMAND` so `ListPlugin`'s own Enter listener runs with whatever options it was configured with (e.g. `shouldPreserveNumbering`), keeping every backward-delete flavor in lockstep with Enter without re-encoding the config; (2) **non-empty nested item** — dispatches `OUTDENT_CONTENT_COMMAND` to outdent one level; (3) **non-empty top-level item** — replaces the `ListItemNode` with a `ParagraphNode` carrying the same inline children, splitting the surrounding list around the cut (mirrors `$handleListInsertParagraph`'s split shape but keeps content). "Empty" means `getChildrenSize() === 0` (anchor IS the ListItemNode); whitespace-only items are deliberately NOT included so per-character Backspace on `-   ` still erodes one space at a time. List items whose children contain a nested `ListNode` are bailed out of — the right rewrite there is structurally ambiguous, default delete handles it. **(B) Ctrl+A range delete** — when the user does Ctrl+A on a doc whose first block is a list, `$removeTextFromCaretRange` preserves the anchor's ListItem (the iteration only flags a node for removal after seeing both its enter and exit carets, and an anchor INSIDE a node never emits the enter caret) and either merges the focus block into it or — when focus is in the SAME block as anchor (the `- aaa` Ctrl+A case) — never merges, leaving an empty `<li>` stub. **Discrimination is provenance-based, not structural**: `- |aaa|` (double-click word-select on "aaa") and `|- aaa|` (Ctrl+A on `- aaa`) produce the EXACT same RangeSelection (anchor=(`aaa`, 0, text), focus=(`aaa`, 3, text)) so the selection alone can't tell them apart — and we want opposite behaviors for them (the word-select case leaves `- ` so the cursor stays where it was, like every other rich-text editor; the Ctrl+A case clears the doc). The plugin therefore hooks `SELECT_ALL_COMMAND` at `COMMAND_PRIORITY_LOW` and stashes a snapshot of the resulting selection's points (key + offset + type for anchor and focus) in a microtask — reading synchronously would capture the pre-`$selectAll` selection since our LOW priority runs before the default mutator at `EDITOR`. It also hooks `SELECTION_CHANGE_COMMAND` at LOW, invalidating the snapshot whenever the live selection no longer matches; the SELECTION_CHANGE that `$selectAll` itself fires runs BEFORE the microtask sets the snapshot so it sees null and harmlessly no-ops, while every later change (caret move, click, history undo) flips it back to null. Each delete handler fires the rewrite only when the live selection still equals the snapshot. The rewrite itself is just `root.clear()` + insert an empty paragraph + `selectStart`; no need to thread through `removeText` once the outcome is known to be "completely empty doc". Hooked on both directions of every range-delete channel — `KEY_BACKSPACE_COMMAND`, `KEY_DELETE_COMMAND`, `DELETE_LINE_COMMAND`, `DELETE_WORD_COMMAND`. Snapshot lives in the `useEffect` closure (mutable `let`), not React state; nothing renders off of it.
- **`NoListInTablePlugin`** — `registerNodeTransform(ListNode)` that unwraps any `ListNode` appearing inside a `TableCellNode` into paragraphs (each `ListItemNode` → `ParagraphNode` carrying its inline children; nested `ListNode`s are lifted into the cell as siblings so the next transform pass unwraps them too). Lists in cells are unworkable in this editor — the cell key surface (Tab navigation, Backspace-deletes-column/-row, in-cell line break) overlaps with the list key surface (Tab to indent, Backspace to exit), and the cell's small visual footprint makes nested list rendering noisy — so we disallow them outright. Markdown shortcut typing (`- ` / `1. ` / `- [ ] `) and on-disk content are blocked **before** a list is ever built by the cell-aware wrappers around `UNORDERED_LIST` / `ORDERED_LIST` / `CHECK_LIST` in `transformers.ts` (the marker stays as literal text instead of being briefly turned into an empty list and then unwrapped — which would erase the marker since a fresh list item has no inline content to preserve). This plugin is the safety net for paths that bypass the transformers (raw `INSERT_*_LIST_COMMAND` dispatches, paste of pre-built nodes, etc.) and handles the structural cleanup uniformly.
- **`CodeBlockEscapePlugin`** — Shift+Enter exits a code block; ArrowUp/ArrowDown escape a block at the document's top/bottom edge.
- **`FloatingFormatToolbarPlugin`** — select text → a "bubble" toolbar fades in above it to toggle bold / italic / strikethrough / inline-code; mouse-first, portaled to `document.body`, buttons `preventDefault` mousedown to keep the selection.
- **`FloatingLinkEditorPlugin`** — Notion-style hover editor: dwell on a manually-authored `[text](url)` link → a panel fades in below it to open / edit / remove the URL; hover- not selection-driven, so every mutation targets the link by node key — `setURL` to edit, unwrap to remove — never `TOGGLE_LINK_COMMAND`; a show/hide grace timer bridges the gap between link and panel; bare-URL `AutoLinkNode`s are excluded since they round-trip as text and are edited by editing their text. Hover is the sole entry point for editing (mouse-first, like the format toolbar — no keyboard / touch path); creation goes through either typing Markdown `[text](url)` or `PasteLinkPlugin`'s selection-and-paste path. Both surfaces are deliberate trade-offs for this desktop editor.
- **`PasteLinkPlugin`** — selection + pasted URL → wrap the selection as `[text](url)` via `$toggleLink`. Hooks `PASTE_COMMAND` at `COMMAND_PRIORITY_LOW` so it preempts the rich-text plugin's `EDITOR` paste (which would replace the selection with the URL text). The URL match is strict — the trimmed clipboard payload must be `https?:` / `mailto:` / `tel:` followed by non-whitespace, with nothing else — so pasting prose that merely contains a URL keeps falling through to the default paste. **Creation-only**: a selection sitting inside any `LinkNode` (manual or autolink) bails, leaving `FloatingLinkEditorPlugin`'s hover panel as the sole URL edit surface — no silent overwrite of an existing link's `__url` when a paste happens on already-linked text. The handler also bails (returns false → default paste runs) on every other shape where the wrap would be wrong: collapsed / whitespace-only selection, any `ElementNode` in `selection.getNodes()` (cross-block selection would create one LinkNode per block), any `LineBreakNode` in `getNodes()` (a soft `Shift+Enter` break wrapped in a LinkNode serializes as `[foo  \nbar](url)`, which the single-line LINK transformer doesn't re-recognize on import — the link silently vaporizes on the next round-trip), and `$isInsideCodeBlock` (a LinkNode inside a `CodeNode` doesn't round-trip — the Markdown serializer emits its contents literally). The code-block and link-ancestor checks read from `$getSelectedTextNodes(selection)` — the boundary-trimmed text-node set — instead of the raw `getNodes()`, since a `getNodes()` zero-width boundary (anchor at end-of-text-node or focus at offset 0) would otherwise mark a touched-but-uncovered link / code block as "inside", bailing the wrap and letting the default paste destroy the intended text outside it.
- **`QuoteNestingShortcutPlugin`** — live conversion of `> ` (or `> > `, `> > > `, ...) typed at the start of a paragraph INSIDE an existing `QuoteNode` into a deeper-nested `QuoteNode`. Required because `@lexical/markdown`'s `MarkdownShortcutPlugin` only runs element transformers at the document root (`$runElementTransformers` bails when the anchor's grandparent isn't `root`), so once the user creates a depth-1 quote and presses Enter to start a new quote line, typing `> ` at the start of that line never reaches the QUOTE regex — the marker would stay as literal text inside the depth-1 quote and nesting would only be reachable via file-load / paste. Mirrors `CheckListShortcutPlugin`'s `registerUpdateListener` + `HISTORY_MERGE_TAG` pattern (same pause-related caveats apply): bails on `HISTORIC_TAG` / `COLLABORATION_TAG` / its own `HISTORY_MERGE_TAG` / `editor.isComposing()`; on every other update reads the freshly-committed state and matches a `ParagraphNode` whose parent is a `QuoteNode`, whose children are exactly ONE `TextNode` (the dirty leaf), and whose text content matches `^((>\s)+)(.*)$`. When matched, the rewrite is re-emitted via `editor.update(rewrite, { tag: HISTORY_MERGE_TAG, discrete: true })` so the convert merges into the just-typed space's history entry — Ctrl+Z reverts to the state BEFORE the marker was typed, never landing on the intermediate "paragraph-with-literal-`> `" state that would re-trigger this same convert on re-application. `markerCount = (matched-prefix-length) / 2` adds that many levels of nesting: typing `> ` once opens one extra level, typing `> > ` (in one go before the first space's convert lands) opens two, and so on. The rewrite builds a fresh nested `QuoteNode` chain `markerCount` levels deep with the matched tail (if any) as the deepest paragraph's content, then `paragraph.replace(root)` swaps the original paragraph for the chain at the same position inside its parent QuoteNode — siblings stay put, so a quote with `> a\n> b` lets the user put the caret at the start of `b`, type `> `, and end up with `> a\n> > b` without disturbing `> a`. Caret lands at offset 0 of the new innermost paragraph via `tailParagraph.select(0, 0)`. Shape gate requires exactly one TextNode child (no LineBreakNode soft break, no inline link / code span), matching `CheckListShortcutPlugin`'s rationale — multiple inline runs are structurally ambiguous about what counts as "tail". No format gate: the marker chars are dropped, the tail is preserved verbatim through a fresh `TextNode` (any format flags carried by the original TextNode are dropped — typing the marker with bold active starts the new nested line as plain text, which matches `CheckListShortcutPlugin`'s behavior).
- **`QuoteExitPlugin`** — owns the two exit-from-quote channels (Enter on an empty trailing line, Backspace at the start of a quote line) at `COMMAND_PRIORITY_LOW` so it preempts rich-text's `EDITOR`-priority defaults. The "Enter stays in the quote on a NON-empty line" half of the contract is NOT in this plugin — it falls out of the `QuoteNode > ParagraphNode > inline` tree shape produced by the nesting-aware QUOTE transformer in `transformers.ts`. `selection.insertParagraph()` walks up to the nearest `INTERNAL_$isBlock` ancestor and calls `insertNewAfter` on it; with the cursor inside a `ParagraphNode` child of a `QuoteNode` that ancestor is the paragraph, so the default split appends a sibling paragraph INSIDE the QuoteNode (no command override). Upstream's flat `QuoteNode > inline` shape would put `QuoteNode` itself as the block ancestor, whose default `insertNewAfter` creates a paragraph AFTER the quote — the long-standing Cork bug where Enter dropped you out of a quote on the first press. With the wrapped structure, this plugin only owns the cases where the default would do the wrong thing. **(A) Enter on empty trailing line.** Triggers when (1) selection is collapsed, (2) nearest paragraph ancestor of the anchor is a direct child of a `QuoteNode`, (3) that paragraph is empty (`getTextContent() === ''`), AND (4) it's the last child of its parent QuoteNode. The rewrite drops the empty paragraph, inserts a fresh paragraph as a sibling AFTER the parent QuoteNode via `insertAfter` (same call lands the cursor in the surrounding QuoteNode for a nested quote or at root for a top-level one — `insertAfter` puts the new node wherever parent sits, no branching needed), then prunes the parent QuoteNode AND any cascade-emptied ancestor QuoteNodes (`> > ` + Enter Enter Enter on an otherwise-empty document would otherwise leave hollow nested `<blockquote>` shells behind). Mid-quote empty paragraphs (with siblings below) keep the default split — pressing Enter there extends the gap, matching how non-trailing empty paragraphs behave at root. **(B) Backward delete at start of quote line.** Hooked across all three backward-delete channels — `KEY_BACKSPACE_COMMAND` (Backspace), `DELETE_LINE_COMMAND` with `isBackward` (Cmd+Backspace on macOS / Ctrl+Backspace on Windows), and `DELETE_WORD_COMMAND` with `isBackward` (Option+Backspace / Ctrl+Backspace inside a word). The structural decision is identical across all three because the only thing they vary on (single char vs to-word-start vs to-line-start) is irrelevant once the caret is at offset 0 of a line — there's nothing within the line to delete, the effect is purely structural. Sharing one body (`$tryExitQuoteAtStart`) means all three keys produce the same quote-exit behavior, matching cross-editor expectations. Lexical's `beforeinput` handler calls `event.preventDefault()` BEFORE dispatching `DELETE_LINE_COMMAND` / `DELETE_WORD_COMMAND`, so those return `true` and no manual preventDefault from us; `KEY_BACKSPACE_COMMAND` IS dispatched without an upstream preventDefault, so the Backspace branch DOES need to call `event.preventDefault()` itself (without it the DOM-level delete races against Lexical's update and the mutation observer can re-sync the editor state to a half-updated DOM, producing structural drift). Triggers when (1) selection is collapsed at `anchor.offset === 0`, (2) the anchor leaf is the paragraph's first descendant (so the caret really is at the line start, not at offset 0 of some non-leading text run), AND (3) the paragraph is a direct child of a QuoteNode. Two branches by paragraph position. **(B1) Empty AND last child** — same outcome as the Enter case: exit downward, paragraph after parent. Without this, Lexical's default Backspace merges the empty paragraph INTO the previous quote line (`> aaa\n> |` → `> aaa|`), reading as "the empty trailing line silently deleted itself with the caret jumping back into the previous line", which is not how every other markdown editor exits a quote on backspace. **(B2) First child** — unwrap upward: move the paragraph BEFORE parent at the grandparent level via `parent.insertBefore(paragraph)`, then prune parent if it emptied. Without this, two failure modes hit at once: (a) the default merges across the QuoteNode boundary into whatever block precedes the quote, so `> aaa\n\n> |bbb` (two adjacent quotes separated by a blank line) collapses into a single `> aaa|bbb` quote with the blank gone — content and structure both mangled. (b) for a single-paragraph quote (`> |aaa`), the default routes through `QuoteNode.collapseAtStart`, which creates a NEW ParagraphNode and re-parents the inner ParagraphNode (the QuoteNode's only child) into it — producing `ParagraphNode > ParagraphNode > TextNode`. That nested-paragraph shape silently breaks `MarkdownShortcutPlugin`'s "grandparent must be root, parent's first child must be a TextNode" gate: typing `> ` after that point never re-triggers the QUOTE shortcut, so the user thinks quote authoring stopped working until they reopen the dialog. Doing the unwrap manually keeps the tree flat — the inner paragraph is detached from the QuoteNode and inserted directly under the QuoteNode's parent — exactly the shape the live shortcut expects. **(B-join) Empty spacer between two adjacent QuoteNodes** — special case that fires BEFORE the QuoteNode parent gate. When the caret sits on an empty paragraph whose previous sibling AND next sibling are both QuoteNodes (typically the shape `$insertSpacersBetweenAdjacentQuotes` restores from a saved `> aaa\n\n> bbb`), Backspace concatenates `next`'s children onto the end of `previous`, drops the spacer paragraph and the emptied `next` shell, and parks the caret at the end of `previous`'s original last paragraph (snapshotted BEFORE the move so it points to the user's `aaa` end, not the freshly-appended `bbb` end). `> aaa\n|\n> bbb` becomes `> aaa|\n> bbb` (one merged QuoteNode containing both paragraphs, caret between them). Without this branch the default delete-character path on a root-level empty paragraph with non-paragraph siblings either no-ops or merges the caret into the leading quote without touching the trailing one — leaving `> bbb` adrift as a separate QuoteNode below.

Past the B-join gate the cursor's paragraph IS a direct child of a QuoteNode, and the entire "eat one `> ` marker off this line" semantic collapses to two branches based on whether the paragraph leads or trails inside its parent — `isEmpty` / `isLast` stop being orthogonal dimensions once "carry the paragraph with the caret" replaces "drop the empty + create fresh". **(B-first) First child** — `$unwrapBefore`: the paragraph moves BEFORE its parent QuoteNode at the grandparent level. Covers `> |aaa\n> bbb` → `|aaa\n> bbb` (leading line escapes upward, trailing siblings stay in the quote) AND the single-paragraph case `> |aaa` → `|aaa` (`$pruneEmptyQuoteAncestors` then removes the emptied parent and cascades up). This branch also sidesteps the default `collapseAtStart` re-parenting bug that wraps the inner paragraph in another paragraph and breaks the live `> ` shortcut afterwards. **(B-extract) Non-first child** — `$extractParagraphDownward`: the paragraph migrates AFTER its parent QuoteNode at the grandparent level, and any trailing siblings inside the original QuoteNode get re-parented into a fresh QuoteNode that sits after the extracted paragraph (so the post-cursor lines keep their quote marker). One call covers four originally-separate shapes — empty-trailing (`> aaa\n> |` → `> aaa\n|`), non-empty-trailing (`> aaa\n> |bbb` → `> aaa\n|bbb`), empty-mid (`> aaa\n> |\n> bbb` → `> aaa\n|\n> bbb`), and non-empty-mid (`> aaa\n> |bbb\n> ccc` → `> aaa\n|bbb\n> ccc`) — once you stop branching on content shape and only ask "are there trailing siblings or not?" (the structural question that decides whether to materialise the trailing QuoteNode). `getNextSiblings()` is snapshotted BEFORE the `paragraph.remove()` so a live walk of the soon-to-move children isn't truncated mid-iteration; the new QuoteNode is inserted FIRST and the extracted paragraph SECOND so `insertAfter`'s "land immediately after the receiver" semantics produce the final order `parent → paragraph → newQuote`. All branches call `event.preventDefault()` before returning true so the browser's native backspace doesn't also fire on the same keystroke — without it, the DOM-level delete races against Lexical's update and the mutation observer can re-sync the editor state to a half-updated DOM, producing structural drift.

- **`HorizontalRuleKeyboardPlugin`** — Up/Down arrows select an adjacent `HorizontalRuleNode` as a node selection instead of skipping it. Lexical's left/right already step onto a rule, but its vertical handlers deliberately leap over block decorators; this plugin restores the symmetry (runs at `COMMAND_PRIORITY_LOW`, above rich-text's `EDITOR`, and bails unless a rule is the next/previous block) so the caret can land on a rule and delete it. Edge detection is geometric — the caret's client rect vs. the block's — so the rule is reachable from _any_ column of the touching visual line (last line for Down, first for Up), not just the block's start/end, while a wrapped paragraph still navigates line-by-line internally.
- **`FormatFormattableTextPlugin`** — owns ranged `FORMAT_TEXT_COMMAND`: formats only prose, never code-block text (which the Markdown serializer would silently drop), and toggles direction by "enable unless every formattable node already has the format", so a mixed selection always enables, consistent with the toolbar's active state, instead of Lexical's first-node-dependent toggle.
- **`FormatShortcutPlugin`** — owns the inline-format Markdown shortcuts (`**bold**` / `*italic*` / `~~strike~~` / `` `code` `` / `==highlight==` / `***bi***` / `___bi___`), replacing the text-format leg of `@lexical/markdown`'s `$runTextFormatTransformers`. The upstream version applies the format via `nextSelection.formatText(format)` after stripping the markers — which TOGGLES based on the first node's current state, so wrapping already-bold text with `**...**` un-bolds it (the pre-check `if (!nextSelection.hasFormat(format))` reads the freshly-created RangeSelection's own format field, always 0, never the underlying text). This plugin keeps upstream's pattern detection verbatim (open/close tag location with sibling traversal, intraword guard, repeating-char disambiguation, code-span precedence, composition-end gating) and replaces only the apply step: walk every formattable text node in the matched range and set the format ON if not already — never off. Mirrors `FormatFormattableTextPlugin`'s set-ON semantics so the shortcut and the toolbar can never disagree. Cohabits with `MarkdownShortcutPlugin` by partitioning the transformer list: the caller passes only the text-format transformers here (`MARKDOWN_TEXT_FORMAT_SHORTCUT_TRANSFORMERS`) and only the rest to `MarkdownShortcutPlugin` (`MARKDOWN_BLOCK_SHORTCUT_TRANSFORMERS`), so upstream's `$runTextFormatTransformers` sees an empty matcher table and no-ops while element / multiline / text-match transformers still run there. Import/export still use the full `MARKDOWN_TRANSFORMERS` list.
- **`CheckListShortcutPlugin`** — live conversion from `- ` + `[ ] ` / `[x] ` (typed sequentially) into a check list. Required because `@lexical/markdown`'s `MarkdownShortcutPlugin` only runs element transformers at the document root (`$runElementTransformers` bails when the anchor's grandparent isn't `root`), so a `[ ] ` typed INSIDE an already-created bullet item never reaches the `CHECK_LIST` regex — leaving the `- ` typed by the user as a bullet item whose body reads literally `[ ] foo`, only to silently flip to a check list across save/reload because `STRICT_CHECK_LIST` in `transformers.ts` matches the exported `- [ ] foo` line. Hooks `registerUpdateListener`: bails on `HISTORIC_TAG` / `COLLABORATION_TAG` / its own `HISTORY_MERGE_TAG` / `editor.isComposing()`; on every other update reads the freshly-committed state and matches a bullet `ListItemNode` whose children are exactly ONE `TextNode` whose content is exactly `[ ] ` / `[x] ` / `[X] ` (4 chars, INCLUDING the trailing space the user just typed), with that TextNode as the dirty leaf. When matched, the rewrite is re-emitted via `editor.update(rewrite, { tag: HISTORY_MERGE_TAG, discrete: true })` so the history plugin MERGES it into the immediately preceding entry (the space-insertion typing) instead of pushing a new one. The undo-stack target is therefore whatever was current BEFORE the trailing space — never the `[ ] ` (4-char) state itself, which is the state that would serialize as `- [ ] ` and re-import as a check item if it survived undo. Two pause caveats: (1) if the user paused past lexical-history's merge-delay (~300ms) between typing `]` and ` `, the prior `[ ]` (3 chars, no trailing space) state was already pushed as its own undo entry and Ctrl+Z lands there — still safe because `- [ ]` doesn't match `STRICT_CHECK_LIST` (the regex requires `\s` after `]`); (2) within the delay window the typing run coalesces and Ctrl+Z reverts the whole run plus the convert. `discrete: true` runs the rewrite synchronously so the `[ ] ` state never paints to a frame. A `KEY_SPACE_COMMAND` + `event.preventDefault()` alternative — converting BEFORE the space inserts — was tried first as the conceptually-cleanest design, but `preventDefault` on the keyboard event passed through `dispatchCommand` didn't reliably suppress the subsequent space insertion in practice, so the typing flow never converted at all. A `registerNodeTransform` alternative ALSO fails: transforms run on `HISTORIC` undo, so Ctrl+Z would restore the bullet-with-`[ ] ` state and the transform would immediately re-convert it — undo unreachable. The update-listener path is the only one that fires reliably AND can short-circuit on history tags. Trigger shape gate: a bullet `ListItemNode` whose children are exactly ONE `TextNode` whose content STARTS WITH `[ ] ` / `[x] ` / `[X] ` (matched via `/^\[([ xX])\] (.*)$/`), AND that TextNode is the dirty leaf for this update — pure caret moves don't dirty leaves so they bail. The trailing capture (`.*`) lets the trigger fire when the user inserts the marker INTO existing text (e.g. caret at start of `aaa`, types `[ ] ` → text becomes `[ ] aaa` → convert preserves `aaa` on the new check item) — not just the empty-bullet typing flow. An item with multiple children (inline link / code span / `LineBreakNode` from Shift+Enter) bails. There is NO format gate: the marker's 4 chars are stripped on convert; the remainder (if any) is preserved verbatim as a fresh `TextNode` on the new check item — so a marker typed with bold/italic active still converts cleanly, no content is lost. The new check ListNode is built as `$copyNode(parentList).setListType("check")` (NOT `$createListNode("check")`) so it inherits parentList's `listMarkerState` — otherwise a `* foo` bullet converted via this shortcut would silently rewrite to `- [ ] foo` on save (`$listExport` writes `${listMarker} [...]` for check lists too, defaulting to `-` only when listMarkerState was never set). Rewrite has three shapes: (1) **Only-item** (top-level OR structurally nested): `parentList.replace(newCheckList)` — the wrapping slot (root or wrapping LI) keeps the same structural position with a check ListNode in place of the bullet one. (2) **Top-level + siblings**: split at `parentList`'s level, mirroring `ListExitPlugin`'s `$splitListAtListItem` — `parentList.insertAfter(newCheckList)`, snapshot `getNextSiblings()` and move them into `$copyNode(parentList)` (NOT `$createListNode('bullet')` — `$copyNode` preserves `listMarkerState`, so a `* foo` list keeps its `*` marker on the trailing items instead of silently becoming `- foo`), remove the original item, remove the original parent list if empty. (3) **Nested + siblings**: can't split at `parentList`'s level — would leave 2-3 sibling ListNodes inside one wrapping LI, breaking `@lexical/list`'s "at most one nested ListNode per LI" invariant and `$listExport`'s nested-recursion guard (`childrenSize === 1 && $isListNode(firstChild)`). Split at the OUTER list's level instead: `$copyNode` the wrapping LI to hold `newCheckList`, insert it after the original wrapping LI; if there are trailing siblings, `$copyNode` another wrapping LI with `$copyNode(parentList)` inside it and move the siblings there. Each wrapping LI ends up holding exactly one nested ListNode → invariant preserved. The resulting tree differs structurally from what the file-import path produces for the same Markdown (import yields two top-level ListNodes for `- foo\n    - bar\n    - [ ] `), but both shapes serialize identically via `$listExport` and render identically — the visible indent depth is driven by the nested-ListNode wrapper, not by which level of the tree holds it. Caret placement: `oldOffset - 4` clamped to `[0, contentLength]` on the carried-over text node — for the typing-empty flow this lands at offset 0 of an empty item, for marker-into-existing-text this lands just before the carried-over content (matching where the caret was after the marker), for paste-with-end-caret this lands at the end of the carried content. `- ` alone is the only path to a check list during live editing: `[ ] ` typed at the document root stays literal text (`STRICT_CHECK_LIST`'s regex requires the `[-*+]\s` prefix and `UNORDERED_LIST` doesn't match `[`), and a saved `[ ] foo` (no dash) round-trips as a plain paragraph — so the on-disk text and the editor's rendering stay in lockstep edit-by-edit.
- **`CheckListOutdentPlugin`** — fully owns `OUTDENT_CONTENT_COMMAND` so type-mismatch outdents preserve the item's `__listType` AND multi-item Shift+Tab works correctly. Upstream `$handleOutdent` (`@lexical/list/formatList.ts`) moves an outdented item into its `greatGrandparentList` without adjusting `__listType`, so a check item Shift+Tab'd through a bullet wrapper lands in the outer bullet ListNode and `ListItemNode.$transform` then clears `__checked` (parent isn't a check list). Multi-item Shift+Tab compounded the bug — every check item in the selection turned into a bullet. Hooks at `COMMAND_PRIORITY_LOW` (ahead of rich-text's EDITOR-priority `$handleIndentAndOutdent` which routes to upstream's `$handleOutdent`). Iterates LIs in the selection (document order, deduplicated, mirroring `@lexical/utils`'s `$handleIndentAndOutdent` block discovery). For each item, applies one of two branches: (1) **Same-type outdent** (`parentList.getListType() === outerList.getListType()`) mirrors upstream verbatim — first/last simple `wrappingLI.insertBefore` / `insertAfter`, middle splits into two new wrapping LIs around the outdented item. (2) **Type-mismatch outdent** lifts the item out of outerList into a fresh `$copyNode(parentList)` ListNode (preserving `__listType`, `__start`, `listMarkerState`) at outerList's parent level. Root case (outerList.parent is root): place `liftedList` as outerList's sibling, split outerList around wrappingLI for siblings — items BEFORE the cut stay in original outerList/parentList; items AFTER move into a `$copyNode(outerList)` after `liftedList` so they keep their original depth. Nested case (outerList.parent is grandWrappingLI): wrap `liftedList` in a new LI alongside grandWrappingLI in grandOuter — handles doubly/deeply nested cases without violating `@lexical/list`'s "at most one nested ListNode per LI" invariant. Cascade cleanup removes empty parentList → wrappingLI → outerList → grandWrappingLI. The structural assumption is the canonical wrap (`wrappingLI.getChildrenSize() === 1 && firstChild === parentList`); non-canonical wrappers (rare — Lexical doesn't normally produce mixed text-plus-nested-list LIs via the indent path) are skipped. The user's caret stays where it was — the item's text node is moved (key preserved) but never re-selected, so a Shift+Tab on `- foo\n  - [ ] bar|` lands the caret on `- foo\n- [ ] bar|` rather than jumping to `- foo\n- [ ] |bar`.
- **`CheckListIndentPlugin`** — symmetric to `CheckListOutdentPlugin`: fully owns `INDENT_CONTENT_COMMAND` for the same reason. Upstream `$handleIndent` unconditionally appends the indented item into `prevSibling.getFirstChild()` (the nested ListNode) when the prev sibling is a wrapping LI — without checking that nested ListNode's type. So a bullet item Tab'd next to a check-containing wrapper silently becomes a check item; multi-item Tab compounded the bug. User-visible bug: `- aaa\n  - [ ] bbb\n- ccc` + Tab on `ccc` produced `- aaa\n  - [ ] bbb\n  - [ ] ccc` (ccc became a check item) instead of `- aaa\n  - [ ] bbb\n  - ccc` (ccc stays a bullet in a NEW nested bullet ListNode next to the check one). Hooks at `COMMAND_PRIORITY_LOW`, iterates selected LIs in document order, applies a unified type-aware indent per item: prefer appending to a SAME-TYPE adjacent nested wrapping LI (prev or next), else create a fresh `$createListItemNode()` wrapping LI containing a `$copyNode(parent)` nested list (preserves `__listType`, `__start`, `listMarkerState`). Matches upstream's structural shape for same-type cases AND solves type-mismatch cases in one path — each item ends up at depth+1 inside a nested ListNode of its OWN type. Multi-item ordering: when the FIRST item creates a new nested wrapping LI, the SECOND item (still in the outer ListNode) sees that wrapping LI as its new prev sibling. Its type-aware indent then appends it to that same nested list (same-type match), giving the natural "indent chain" — consecutive selected items merge into the same nested list, matching upstream's multi-item shape.

Lexical's **`AutoLinkPlugin`** wraps bare `https://` / `http://` URLs (typed or loaded from file) in `AutoLinkNode`s so they're clickable without `[text](url)` syntax; the Markdown serializer skips `AutoLinkNode`, so URLs round-trip as raw text and the file is never rewritten to `[url](url)`; fenced code blocks are excluded via `excludeParents`. Lexical's **`ListPlugin`** supplies empty-item Enter-to-exit (every backward-delete flavor's exit, plus the non-empty-item-start outdent / paragraph-conversion, is added by `ListExitPlugin` above). Lexical's **`CheckListPlugin`** wires GitHub-style task lists: click toggles a checkbox, Space toggles a focused check item, ArrowUp/Down navigate between check items, Esc returns focus to the editor root, and `INSERT_CHECK_LIST_COMMAND` converts the current selection's paragraph / list into a check list (`$insertList('check')`). All pointer handlers are zone-gated — they read `getComputedStyle(li, '::before').width` and only fire when the click X falls inside the checkbox's pseudo-element — so a click on a link or in the text body still routes to `LinkOpenPlugin` / normal caret placement. The checkbox itself is drawn by the `cork-checklist-item` styles (see `style.css`), not a real `<input>`; the plugin only adds `role="checkbox"` / `aria-checked` attributes on leaf check items for screen readers. Round-trip support lives in `transformers.ts` — `STRICT_CHECK_LIST` (upstream `CHECK_LIST` with its outer `(?:[-*+]\s)?\s?` tightened to a required single `[-*+]\s`, AND its inner `(\s|x)?` tightened to a required `(\s|x)`) is ordered AHEAD of `UNORDERED_LIST` / `ORDERED_LIST` so the first-match-wins import path (`$importBlocks` in `@lexical/markdown`) reads `- [ ] task` as a check list, not a bullet item with text `[ ] task`. The exact single-space marker prefix is the lockstep guarantee — `$listExport` writes `${listMarker} [${checked ? 'x' : ' '}] ` for check lists (using the listNode's `listMarkerState`, default `-`), so accepting any other shape on import would silently normalize the file: a `[ ] task` (no dash) or `-  [ ] task` (double space) or `- [] task` (empty brackets) line stays literal text in the editor AND on disk, where the canonical `- [ ] task` reads as a check item. GFM accepts `-` / `*` / `+` as task list markers (the marker is just the underlying list item's bullet — see https://github.github.com/gfm/#task-list-items-extension-); we accept all three, and `listReplace` propagates `match[0].trim()[0]` onto the new check list's `listMarkerState` so `* [ ] task` / `+ [ ] task` round-trip with their original markers intact. `- [ ] ` typed character-by-character can't reach `STRICT_CHECK_LIST` via the shortcut path (the `- ` triggers `UNORDERED_LIST` first, then `$runElementTransformers` bails inside the new list because the anchor's grandparent isn't root) — `CheckListShortcutPlugin` (above) handles the in-list-item conversion path so the rendering stays consistent with the saved file edit by edit. The check item's `__checked` state round-trips through the file as `- [x] ` / `- [ ] `.

## Quote blocks

Markdown blockquotes (`> `) round-trip through a Cork-owned QUOTE element transformer in `transformers.ts` (registered ahead of `@lexical/markdown`'s flat default, which is filtered out of the merged list). Two reasons we replace the upstream version:

1. **Nesting**. Upstream `^>\s/` matches a single `> ` and treats everything after as inline text, so `> foo\n> > bar\n> > > baz` flattens into one `QuoteNode` whose content reads `foo\n> bar\n> > baz` — the deeper `>` markers survive as literal text instead of rendering as nested quotes. Our regex is `^(>\s)+/` and counts depth as `match[0].length / 2`; each `>` level becomes a nested `QuoteNode`, so the tree mirrors what CommonMark says the source means and nested `<blockquote>` elements visibly indent each level.
2. **Enter behaviour**. Default `QuoteNode.insertNewAfter` returns a fresh `ParagraphNode` appended AFTER the quote — so Enter inside upstream's flat `QuoteNode > inline` shape immediately exits the block. Our transformer wraps each quote line in a `ParagraphNode` child, which makes the paragraph the cursor's nearest `INTERNAL_$isBlock` ancestor; the default split then drops a sibling paragraph INSIDE the QuoteNode and the user stays in the quote (no command override needed for the non-empty case — only `QuoteEnterPlugin`'s exit case requires a hook).

### Tree shape

A `QuoteNode` contains a mix of `ParagraphNode` children (each = one logical quote line at this depth) and nested `QuoteNode` children (= deeper-level quotes interspersed with this depth's lines, in document order). `> foo\n> > bar\n> > > baz` is

```
QuoteNode
  ParagraphNode "foo"
  QuoteNode
    ParagraphNode "bar"
    QuoteNode
      ParagraphNode "baz"
```

and `> foo\n> > bar\n> baz` (drop back to depth 1 after a depth-2 line) is

```
QuoteNode
  ParagraphNode "foo"
  QuoteNode
    ParagraphNode "bar"
  ParagraphNode "baz"
```

so the document-order interleave of paragraphs and nested QuoteNodes preserves where the source line "returned to a shallower level". Same-depth consecutive lines append as separate `ParagraphNode` siblings at the same level rather than merging into one paragraph with `LineBreakNode` separators — both shapes serialize identically through the recursive export, so picking the simpler one (one paragraph per source line) keeps the merge logic flat. CSS makes them read as continuous lines anyway (`.cork-quote p { margin: 0 }` in `style.css`).

### Import (`replace`)

For each `> > > ` line:

- Wrap the inline content (the post-prefix `children`) in a fresh `ParagraphNode`.
- If the previous root sibling is a `QuoteNode`, splice this paragraph into its tree via `$mergeIntoQuoteTree`: walk the previous quote's `getLastChild()` path down through nested QuoteNodes to find the "current tail depth", then either (a) append to that level if depths match, (b) open additional `QuoteNode` levels from the tail if our line is deeper, or (c) re-descend from the outer QuoteNode only to our line's depth if our line is shallower. The freshly-imported paragraph always lands at exactly `targetDepth`.
- Otherwise build a fresh nested `QuoteNode` chain `depth - 1` levels deep, append the paragraph inside the innermost, and `parentNode.replace` with the outer.
- Live typing (`!isImport`) skips the merge step and sets the caret to the new paragraph via `paragraph.select(0, 0)`.

### Export

`$exportNestedQuote(quote, exportChildren, depth)` walks the tree recursively. For each child:

- `QuoteNode` → recurse with `depth + 1` and concat the resulting line list.
- Any other `ElementNode` (in practice a `ParagraphNode`) → call `exportChildren(child)` to get its inline markdown (so bold / italic / code / links / autolinks run through the standard transformer pipeline), `split('\n')` on the result (each `\n` came from a `LineBreakNode`), and prefix every line with `'> '.repeat(depth)`.

An empty `QuoteNode` (no children — defensive) emits a single line of `prefix.trimEnd()` so it doesn't dangle trailing whitespace, but in normal operation every QuoteNode has at least one paragraph child.

### Adjacent QuoteNode spacer (post-import)

`@lexical/markdown`'s `createMarkdownImport` strips empty paragraphs at root after the line-by-line pass (the `isEmptyParagraph` cleanup loop), so a `> aaa\n\n> bbb` source — which the line walker initially built as `[QuoteNode "aaa", emptyParagraph, QuoteNode "bbb"]` — collapses to `[QuoteNode "aaa", QuoteNode "bbb"]` by the time the import returns. The two adjacent blockquotes then render flush against each other, while the live editor (where the user authored them with an Enter-exit + blank line + `> bbb`) had the empty paragraph in between as a visible gap. `$insertSpacersBetweenAdjacentQuotes` runs immediately after `$convertFromMarkdownString` to restore that gap so the post-load shape matches the post-author shape, edit by edit. The walk caches `getNextSiblings()`-equivalent state (the captured `next` reference) BEFORE inserting the spacer, so the iteration lands on the original next sibling rather than the freshly-inserted spacer (preventing a no-op re-check and reading more obviously as "walk the original list"). Only fires at root level — nested adjacent QuoteNodes don't occur in our tree (consecutive same-depth quote lines merge into one QuoteNode with two paragraphs via `$mergeIntoQuoteTree`), and the `.cork-quote p { margin: 0 }` rule would zero any spacer margin we tried to insert there anyway.

### Live typing

`MarkdownShortcutPlugin` only runs element transformers when the anchor's grandparent is `root` (a single-line text in a top-level paragraph), so the QUOTE element-transformer's shortcut path only fires at root level. Typing `> ` at the start of an empty root paragraph creates a depth-1 quote and parks the caret in the inner paragraph; typing `> > ` at the start of a fresh root paragraph (in one go, before the first space fires the convert) creates a depth-2 nested structure. Once the caret is INSIDE a `QuoteNode`, the grandparent-is-root gate fails — that's where `QuoteNestingShortcutPlugin` (above) takes over: it watches `registerUpdateListener` for a paragraph inside a QuoteNode whose only TextNode child just became `> ` / `> > ` / ... and converts it to a deeper-nested QuoteNode chain in lockstep with the typing history (HISTORY_MERGE_TAG). So the user can type `> foo` (depth-1) → Enter → `> bar` and the second `> ` opens a depth-2 nested quote with cursor parked inside; the typing path matches the on-disk import path, edit by edit.

The `replace` callback ALWAYS checks for an immediately-adjacent QuoteNode as the previous sibling and merges into it (via the same `$mergeIntoQuoteTree` the import path uses) — regardless of `isImport`. The import case is the obvious one (consecutive `> ` lines), but the typing case matters too: after the user exits a quote (Enter on an empty trailing line → fresh paragraph at root after the QuoteNode) and then types `> bbb` on that fresh paragraph, they expect the new line to join the previous quote (`> aaa\n> bbb`), not start a second QuoteNode (`> aaa\n\n> bbb`). Same call site handles both — the previous-sibling check only fires when the paragraph the shortcut is replacing is genuinely a root sibling of the existing QuoteNode, so a paragraph that sits between the QuoteNode and the cursor (e.g. text content typed after exit) keeps the create-fresh path.

In live mode the same `replace` also runs `$absorbTrailingQuoteSibling` after the previous-merge (or after building a fresh chain when the previous sibling isn't a QuoteNode): if the resulting top-level QuoteNode now has a QuoteNode as ITS next sibling, the trailing QuoteNode's children get appended onto the merged one and the trailing shell is dropped. This bridges the spacer-paragraph case — typing `> bbb` on the empty paragraph in `> aaa\n|\n> ccc` (the exact shape `$insertSpacersBetweenAdjacentQuotes` restores from a saved `> aaa\n\n> ccc`) folds all three lines into a single `> aaa\n> bbb\n> ccc` quote instead of leaving the trailing `> ccc` adrift. Gated on `!isImport && depth === 1`: the import path's line-by-line previous-merge already absorbs adjacent same-depth lines, and depth>1 typing can't reach this code path anyway (`MarkdownShortcutPlugin`'s grandparent-is-root gate fails inside an existing QuoteNode, so a multi-level `>` chain only reaches `replace` via paste / import, both of which are `isImport`).

## Horizontal rules

Lexical's `HorizontalRulePlugin` + built-in `HorizontalRuleNode` (an `<hr>` decorator) render thematic breaks. A rule is authored by typing `---`, `***`, or `___` on its own line (the `HORIZONTAL_RULE` element transformer in `transformers.ts` is `triggerOnEnter`); `@lexical/markdown`'s defaults have none, so we add it. Import accepts all three CommonMark markers (3+ chars, non-spaced only — a spaced `- - -` would collide with an unordered-list item); export always writes `---`. This means only `---` round-trips byte-for-byte: a file that used `***` or `___` has its rules **normalized to `---`** the first time the body is edited and saved (a deliberate trade-off — `HorizontalRuleNode` doesn't carry which marker it came from — unlike tables, which preserve their input shape). The transformer also bails inside table cells (`$getTableCellNodeFromLexicalNode`, mirroring `TABLE`), so a cell body of literally `---`/`***`/`___` stays text instead of nesting an `<hr>` in the cell. Styling lives in `style.css` (`cork-hr` draws the line + click target via `::after`, `cork-hr-selected` outlines the selected rule) and is wired through the editor theme's `hr` / `hrSelected` keys; the rule is selectable (and thus deletable) by click, by left/right caret, and — via `HorizontalRuleKeyboardPlugin` — by up/down caret.

## Tables

Lexical's `TablePlugin` enables `TableNode` (cell selection, Tab navigation, `INSERT_TABLE_COMMAND`). Lists are disallowed inside cells because the cell key surface (Tab navigation, Backspace-deletes-column/-row, in-cell line break) collides with the list key surface (Tab to indent, Backspace to exit) and nested lists don't render legibly in a cell's footprint. Enforcement is two-layered: the `UNORDERED_LIST` / `ORDERED_LIST` / `CHECK_LIST` transformers in `transformers.ts` are `cellAware`-wrapped so a typed `- ` / `1. ` / `- [ ] ` inside a cell stays as literal text (the marker is restored on import too, mirroring the `TABLE` / `HORIZONTAL_RULE` guards); `NoListInTablePlugin` then unwraps any `ListNode` that still slips into a cell via a non-transformer path (raw command dispatch, paste of pre-built nodes). Cell bodies of literally `---` / `***` / `___` likewise stay text instead of nesting a rule (see the `HORIZONTAL_RULE` cell guard).

### Creation & round-tripping

- A table is created by typing one GFM pipe row (`| a |`) and committing it — the `TABLE` transformer is `triggerOnEnter` so both Space and Enter commit, and the trailing-whitespace of `match[0]` tells them apart: **Space** keeps building the header (promotes the row to a header and appends an empty header column, caret parked there to name the next column), **Enter** finishes the header (promotes it and adds an empty body row, caret in the first body cell).
- On _import_ (`isImport`) the source's literal shape is kept instead (a bare row stays header-less until the source divider promotes it, body rows arrive as their own lines), so files round-trip unchanged. The divider regex requires `-+` per column (not `-*`) so a blank `|  |` row isn't mistaken for a divider on reload.
- Cells are trimmed on creation (`| aaa |` → `aaa`) and their bodies are escaped on export / unescaped on import (`encodeCell`/`decodeCell`: `\`, `|` and newlines → `\\`/`\|`/`\n`) so a `|` inside a cell isn't read as a column boundary on reload; import splits rows on unescaped pipes only (`/(?<!\\)\|/`).
- Export emits the `| --- |` divider after the first row unconditionally (not per `__headerState`) so the output is always valid GFM (delimiter on line 2, exactly one header).
- The `TABLE` transformer bails (returns false) when its matched row sits inside a table cell (`$getTableCellNodeFromLexicalNode(parentNode)`), so a cell whose text looks like a pipe row — typed live or restored from a cell body on import — stays literal text instead of nesting a table. On import the guard also restores the line text (`$importBlocks` slices the matched text off the node before calling `replace` and doesn't roll back on a false return), or a cell body like `| x |` would reload empty.

### `TableKeyboardPlugin` (editing surface)

Keyboard-only, no toolbar — the table grows/shrinks under the keys (all handlers at `COMMAND_PRIORITY_CRITICAL` so they win over TablePlugin's built-in HIGH Tab/delete handlers, and each returns false the instant the caret isn't in its situation):

- **Tab** on the rightmost cell adds a column to the right and steps into it (every other Tab is left to the built-in cell navigation).
- **Enter** adds a row below and moves into it; **Shift+Enter** inserts an in-cell line break; **Enter** on a trailing empty (non-header) row exits below the table, dropping that row.
- **ArrowDown/ArrowUp** keeps moving within a multi-line cell (returns `true` without `preventDefault` so the browser does the native line move, blocking the built-in whose rect heuristic mis-jumps rows from the line after an in-cell break), and on the cell's edge line escapes into a fresh paragraph below/above when the table is the document's last/first block (edge detection is structural via `$hasLineAboveInCell`/`$hasLineBelowInCell`, not rects).
- **ArrowUp** from the block directly below a table (on its first line) enters the last row's _first_ cell rather than wherever native caret-x lands (which jumps to the right edge).
- **Backspace** in an empty cell deletes the column when the whole column is empty (highest-priority, guarded so the last column never goes this way), else jumps to the end of the cell on its left, else in the empty leftmost cell of an empty row deletes that row — a header row is protected (irreversible) unless no body rows remain (`$hasBodyRow`), so the table can still be fully deleted — removing the whole table when it was the last row.
- **Cmd/Ctrl+Backspace** (DELETE_LINE) is reimplemented for cells since TablePlugin swallows it (delete-to-line-start via `modify` + `removeText`, omitting the cell-escaping `deleteCharacter` fallback).

### Layout & selection

- A wide table scrolls horizontally inside a wrapper div (TablePlugin `hasHorizontalScroll` → `theme.tableScrollableWrapper` = `max-w-full overflow-x-auto pb-2`; editor root carries `min-w-0`) rather than widening the dialog — the wrapper, not the `<table>`, because `border-collapse` tables ignore padding, and the wrapper's `pb-2` reserves a strip so the horizontal scrollbar doesn't overlap the last row.
- A `SELECTION_CHANGE` handler (in `TableKeyboardPlugin`) keeps the caret's cell horizontally in view — moving into an off-screen column via Tab / arrows / typing scrolls the wrapper's `scrollLeft` to follow (rAF-deferred so a just-inserted column is laid out first; only the wrapper scrolls, never the page).
- Cross-cell (grid) selection is Lexical's built-in drag-select; it's made visible via the `tableCellSelected`/`tableSelection` theme classes (`cork-table-cell-selected` overlay + `cork-table-grid-selection` native-selection suppression, both in `style.css`) — without them the selection works but shows no highlight.

### Transformers

The default `@lexical/markdown` transformers have no table or horizontal-rule support, so `transformers.ts` exports `MARKDOWN_TRANSFORMERS` = a GFM `TABLE` element transformer (round-trips `TableNode` ⇄ pipe-delimited Markdown; cell bodies recurse through the same list, hard newlines collapse to literal `\n`) plus a `HORIZONTAL_RULE` element transformer (`---`/`***`/`___` ⇄ `HorizontalRuleNode`), both prepended to the defaults; all three of import / export / `MarkdownShortcutPlugin` use it.

## Testing

Specs live next to their target as `*.spec.ts` / `*.spec.tsx`. Shared helpers are in `__tests__/utils.tsx` — go through them so every test starts from the production-shaped editor (`MarkdownEditor.tsx` exports `NODES` and `buildInitialConfig`; the test helpers import both directly, so node set + theme + state-seed pipeline always match production).

Five test-shape recipes cover what the editor exposes today.

| Shape                      | Example                                              | When to use                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| -------------------------- | ---------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Pure `$`-helper            | `codeBlock.spec.ts`                                  | Pure structural / tree-walk helpers (`$isInsideCodeBlock`, `$isFormattableTextNode`, ...). Wrap with `createTestHeadlessEditor()` — no React, no DOM mount.                                                                                                                                                                                                                                                                                                                                                  |
| Transformer round-trip     | `transformers.spec.ts`, `transformers.quote.spec.ts` | Import/export a Markdown string through `MARKDOWN_TRANSFORMERS`. `$setMarkdown` + `$readMarkdown` on a headless editor.                                                                                                                                                                                                                                                                                                                                                                                      |
| Plugin keyboard contract   | `QuoteExitPlugin.spec.tsx`                           | A plugin gated on a key press. Mount via `renderTestEditor({plugins: <YourPlugin />})` and dispatch via `dispatchKeyDown(editor, "Backspace")`. The helper wraps the dispatch in a discrete `editor.update` so the listener's mutations commit synchronously — Lexical's `triggerCommandListeners` defers commits to a microtask otherwise, and the test's follow-up `read` would see the old state.                                                                                                         |

Distinct from the transformer round-trip shape — that one is headless and never paints.                                                                                                                        |

Asserting rendered DOM: prefer the structural triple (`editorRoot.children` length + `firstElementChild.tagName` + `firstElementChild.textContent`) over `toBeInTheDocument()` / `toBeVisible()` alone. The matchers pass even when an extra paragraph / stray node renders alongside the expected one; the triple pins the rendered shape exactly without freezing theme classes or Lexical-internal attributes (`dir`, `data-lexical-text`, ...) that an `innerHTML` snapshot would lock down and that a class rename would falsely flag.

Import conventions (NOT interchangeable with Testing Library):

- `render` from `vitest-browser-react` — returns retry-able `Locator`s integrated with `expect.element`. `await render(...)` (it resolves after the first effect flush).
- `page` / `userEvent` from `vitest/browser` — `userEvent` dispatches real Chrome DevTools Protocol events.
- `expect.element(…).toBeInTheDocument()` / `.toBeVisible()` / etc. are built into `@vitest/browser` (27 matchers, auto-registered) — do NOT import from `@testing-library/*`. Testing Library's `render` returns DOM elements (no auto-retry), its `userEvent` fakes events, and its cleanup runs in `afterEach` instead of `beforeEach` — the two stacks have different semantics.
